# AML-Graph простыми словами — гид для тех, кто не работал с ML

Этот документ объясняет проект «с нуля»: что он делает, зачем, и что значат все
страшные слова. Никаких предварительных знаний не нужно.

---

## 1. О чём проект — в трёх абзацах

Представьте банк. Каждую секунду люди переводят друг другу деньги. Среди миллионов
честных переводов прячутся **мошеннические цепочки**: преступники «отмывают» деньги —
прогоняют их через десятки счетов, чтобы скрыть, откуда они взялись.

Банк обязан такие цепочки находить и сообщать о них государству. Для этого стоят
автоматические «сторожа»-правила, но они устроены примитивно: «перевод больше 600 000 ₽?
— бей тревогу!». В итоге **9 из 10 тревог — ложные**, а живой сотрудник (аналитик)
тонет в их разборе. Хуже того: даже если система права, она не умеет объяснить,
*почему* перевод подозрительный, — а без объяснения отчёт государству не напишешь.

**Наш проект решает обе проблемы.** Он смотрит не на отдельные переводы, а на
**узор, который переводы рисуют** (кто → кому → когда → сколько), находит известные
«узоры отмывания», и — главное — **сам пишет объяснение** человеческим языком:
«Счёт X разослал деньги на 10 счетов за 2 часа, потом всё собралось на счёте Y —
похоже на схему такую-то, рекомендуем проверить».

## 2. Словарик: все аббревиатуры и термины

### Про предметную область (банки и отмывание)

| Термин | Как читается | Что значит по-простому |
|---|---|---|
| **AML** | Anti-Money Laundering | «Противодействие отмыванию денег» — всё, что банки делают, чтобы ловить грязные деньги. Отсюда название проекта |
| **SAR** | Suspicious Activity Report | Официальный отчёт «мы заметили подозрительное», который банк отправляет государству. Чтобы его написать, нужно **объяснение**, а не просто «модель сказала 0.97» |
| **False positive (FP)** | ложное срабатывание | Тревога, которая оказалась пустышкой. Их сейчас 90–95 % — главная боль аналитиков |
| **Дроп** | — | Подставной человек, на чей счёт выводят краденое. «Fan-out на дропов» = разослать деньги веером по таким счетам |
| **Комплаенс** | compliance | Отдел банка, который следит за соблюдением законов. Наш «заказчик» |

### Про данные и граф

| Термин | Что значит по-простому |
|---|---|
| **Граф** | Картинка из кружков и стрелок. У нас кружок = банковский счёт, стрелка = перевод денег. Миллионы переводов = огромная паутина |
| **Узел / ребро** | Кружок / стрелка на графе. «Ребро» — просто математическое название стрелки |
| **Темпоральный** | «Учитывающий время». Каждая стрелка помнит, *когда* был перевод — для отмывания порядок событий критичен (сначала собрал, потом раздал) |
| **Мультиграф** | Граф, где между двумя кружками может быть несколько стрелок (Вася переводил Пете трижды — три стрелки) |
| **Мотив** | Узнаваемый узор на графе. Как фигуры в вышивке: посмотрел — узнал. Наши мотивы: **fan-out** (один разослал многим), **fan-in** (многие скинулись одному), **cycle** (деньги прошли по кругу и вернулись), **gather-scatter** (собрал → раздал), **scatter-gather** (раздал → собрали обратно) |
| **IBM AMLworld** | Наш набор данных: 5 миллионов *сгенерированных* (не настоящих!) банковских переводов, где исследователи IBM честно пометили, какие переводы — отмывание и по какой схеме. Опубликован на площадке Kaggle |
| **Kaggle** | Сайт-хранилище открытых наборов данных + соревнования по анализу данных |
| **Elliptic** | Второй известный набор данных — транзакции биткоина с пометками «незаконная/законная» |
| **Ground truth** | «Правильные ответы» — эталонная разметка, с которой мы сравниваем догадки модели |

### Про машинное обучение

| Термин | Что значит по-простому |
|---|---|
| **ML** (Machine Learning) | Машинное обучение: вместо того чтобы программировать правила вручную, мы показываем компьютеру тысячи примеров «вот это отмывание, а это нет», и он **сам** находит закономерности |
| **Модель** | Результат обучения — «натренированный нюх». Ей показываешь новый перевод, она отвечает числом от 0 до 1: насколько похоже на отмывание |
| **Признак** (feature) | Число, описывающее объект, — то, «на что смотрит» модель. Наши признаки для перевода: скольким счетам отправитель разослал деньги за сутки, состоит ли перевод в кольце, необычно ли велика сумма и т. п. |
| **Обучение / train** | Процесс «показывания примеров» модели |
| **Тест / test** | Экзамен: проверяем модель на примерах, которых она **не видела** при обучении. Иначе это как дать студенту на экзамене те же задачи, что он зубрил, — оценка ничего не скажет |
| **Валидация / validation** | «Пробный экзамен» между обучением и настоящим тестом. На нём мы подкручиваем настройки. Трогать настоящий тест для настройки — жульничество |
| **Утечка (leakage)** | Главный грех в ML: когда в обучение случайно просочилась информация «из будущего» или из теста. Модель кажется гениальной, а в бою проваливается. У нас от этого защищает **временно́е разбиение**: учимся на ранних переводах, экзаменуем на поздних — как в жизни |
| **Дисбаланс классов** | Когда одного типа примеров кратно больше: у нас на 1000 переводов лишь 1 — отмывание. Из-за этого обычные способы оценки врут (см. раздел 5) |
| **Порог (threshold)** | Модель выдаёт число 0…1, а решение нужно «да/нет». Порог — планка: выше — тревога, ниже — пропускаем |
| **GBDT / LightGBM** | Семейство моделей «градиентный бустинг» — команда из сотен маленьких решающих деревьев, каждое чуть-чуть исправляет ошибки предыдущих. **LightGBM** — конкретная быстрая реализация от Microsoft. В нашем проекте это «рабочая лошадка», и она пока сильнее нейросети |
| **Нейросеть** | Модель, устроенная по мотивам мозга: слои «нейронов», каждый — простая формула. Хороша, когда закономерности сложные |
| **GNN** (Graph Neural Network) | Графовая нейросеть — нейросеть, которая умеет «ходить по стрелкам»: чтобы оценить счёт, она смотрит на его соседей, соседей соседей и т. д. **GraphSAGE** — конкретная разновидность, которую используем мы |
| **LLM** (Large Language Model) | Большая языковая модель — нейросеть, умеющая писать текст (как ChatGPT или Claude). У нас она пишет объяснение-нарратив для аналитика |
| **Нарратив** | Связный текст-объяснение: «что случилось, почему подозрительно, что делать» |

### Про оценку качества (метрики)

| Термин | Что значит по-простому |
|---|---|
| **Метрика** | Число-оценка: насколько хорошо работает модель |
| **Accuracy** | «Доля правильных ответов». Звучит хорошо, но при дисбалансе **обманывает** (см. раздел 5) — поэтому мы её не используем |
| **Precision** (точность) | Из всех, на кого модель «показала пальцем», сколько действительно виновных? Precision 1.0 = ни одного ложного обвинения |
| **Recall** (полнота) | Из всех реально виновных скольких модель поймала? Recall 0.5 = поймала половину |
| **F1** | Одно число, объединяющее precision и recall (их «среднее гармоническое»). **Minority-F1** — F1, посчитанный именно по редкому классу (отмывание) |
| **precision@k** | Точность в топ-k: если аналитик успеет разобрать только 100 самых подозрительных тревог, сколько из них будут настоящими? У нас **precision@100 = 1.00** — все 100 настоящие |
| **PR-AUC** | Площадь под кривой «precision против recall» — оценка модели сразу при всех порогах, а не при одном. Чем ближе к 1, тем лучше; у случайного гадания ≈ доля редкого класса (у нас 0.03) |
| **Alert-reduction** | Наша «бизнес-метрика»: на сколько процентов сократится очередь тревог у аналитика, если он будет доверять модели. У нас **91 %**: вместо 100 бумажек разобрать 9 |

### Про инженерные инструменты

| Термин | Что значит по-простому |
|---|---|
| **Git** | «Машина времени» для кода: хранит каждую версию, можно откатиться и посмотреть, кто что менял. **GitHub** — сайт, где этот архив лежит онлайн |
| **Коммит** | Одно сохранение в git с подписью «что сделано» |
| **CI** (Continuous Integration) | Робот-контролёр: при каждом изменении кода сам запускает проверки (тесты, чистоту стиля). Не прошёл — сразу видно. У нас — GitHub Actions |
| **Тесты (pytest)** | Маленькие программки, проверяющие большие: «если подать на вход кольцо из 3 счетов — детектор обязан его найти». У нас 16 тестов |
| **Docker** | «Коробка», в которую упакована программа со всем окружением. На любой машине запускается одинаково: `docker compose up` — и сервис работает |
| **DVC** (Data Version Control) | Как git, но для больших данных: помнит, из каких данных какой результат получен, сами гигабайты в git не кладёт |
| **MLflow** | Журнал экспериментов: какие настройки → какие метрики. Чтобы через месяц вспомнить, что уже пробовали |
| **API** | Дверь, через которую одна программа обращается к другой. **FastAPI** — инструмент, чтобы такую дверь построить. Наш API: «дай счёт → верну цепочку с оценками и объяснение» |
| **Streamlit** | Инструмент для быстрых веб-интерфейсов на Python. Наше демо: выпадающий список счетов + картинка графа |
| **Ruff** | Автопроверка аккуратности кода (форматирование, неиспользуемые куски) |

## 3. Как машина «учится» — на пальцах

Обычная программа — это инструкция, написанная человеком: «если сумма > X, тревога».
Машинное обучение переворачивает подход: **мы не пишем правило — мы показываем примеры.**

1. Берём 175 000 старых переводов, где уже известно, какие были отмыванием.
2. Для каждого считаем признаки-подсказки: «скольким людям отправитель разослал
   деньги за сутки?», «вернулись ли деньги по кругу?», «необычна ли сумма?».
3. Модель перебирает эти подсказки и **сама** находит сочетания, отличающие
   отмывание от нормы. Это и есть «обучение».
4. Потом — экзамен на 75 000 **новых** переводах (из будущего периода, модель их
   не видела): для каждого модель выдаёт оценку риска от 0 до 1.

У нас таких «учеников» два: **LightGBM** (команда решающих деревьев — быстрая и
точная) и **GNN** (нейросеть, умеющая ходить по стрелкам графа). Спойлер: пока
побеждает LightGBM — и это нормальный, честный результат (см. раздел 6).

## 4. Путь данных: что происходит от начала до конца

```
CSV с 5 млн переводов
      │  (1) читаем, берём подвыборку 250 тыс. — но ВСЕ примеры отмывания сохраняем
      ▼
Граф: 200 тыс. счетов-кружков, 250 тыс. стрелок-переводов
      │  (2) ищем на графе мотивы: веера, кольца, «собрал-раздал»
      ▼
Таблица признаков: каждая стрелка описана числами
      │  (3) делим ПО ВРЕМЕНИ: старое — учёба, среднее — настройка, новое — экзамен
      ▼
Две модели учатся → каждая ставит оценку риска каждой стрелке экзамена
      │  (4) считаем честные метрики, строим графики
      ▼
Берём конкретную найденную схему (кольцо из 10 счетов)
      │  (5) подсвечиваем цепочку на картинке
      ▼
LLM пишет нарратив: «10 счетов, $2.28 млн за 88 часов, кольцо, рекомендация — проверка»
      │  (6) всё это доступно в веб-демо и через API
      ▼
Аналитик видит картинку + объяснение → готовая заготовка для отчёта SAR
```

## 5. Почему мы не используем «долю правильных ответов»

Представьте: из 1000 переводов только 1 — отмывание. Возьмём «модель»-лентяйку,
которая **всегда** отвечает «всё чисто». Её accuracy = 999/1000 = **99.9 %**.
Выглядит блестяще — а пользы ноль: единственного преступника она пропустила.

Поэтому при редком классе смотрят на другое:

- **Precision** — не позорим ли мы невиновных? (доля настоящих среди обвинённых)
- **Recall** — скольких преступников поймали из всех?
- **F1** — баланс первых двух одним числом.
- **precision@100** — самое житейское: аналитик физически разберёт ~100 тревог в день;
  важно, чтобы **верх списка** был чистым. У нас все 100 из топ-100 — настоящие.
- **Alert-reduction** — сколько работы сэкономили: у нас при согласии пропустить
  половину схем (recall 0.5) очередь тревог сокращается на **91 %**.

И второй столп честности — **разбиение по времени**. Мы никогда не подглядываем в
будущее: учимся на переводах «до 12 сентября», экзаменуем на «после». Именно так
модель будет жить в бою. Специальный тест в коде роняет сборку, если кто-то
случайно нарушит это правило.

```

  data_io.py      скачивание данных и чтение «правильных ответов»
  graph_build.py  строит граф из таблицы переводов

docs/       ← документы (вы читаете один из них)
outputs/    ← результаты: 6 картинок + metrics.json с числами
data/       ← сюда скачиваются данные (в git не попадают — слишком большие)
Dockerfile, docker-compose.yml  ← «коробки» для запуска одной командой
.github/workflows/ci.yml        ← робот-контролёр CI
```

## 8. Частые вопросы

**— Это настоящие банковские данные?**
Нет. IBM AMLworld — реалистичная симуляция: искусственный «городок» из агентов,
которые переводят друг другу деньги, и часть из них по сценарию отмывает. Зато
у каждого перевода есть честная пометка — на настоящих данных такой роскоши нет.

**— Модель сама решает, кого блокировать?**
Нет. Она только **сортирует очередь** для человека и объясняет свои подозрения.
Решение всегда за аналитиком — так и требуют регуляторы.

**— Почему подозрительных 2–3 %, если в жизни 0.1 %?**
Мы работаем на подвыборке и специально сохранили все примеры отмывания, чтобы
моделям было на чём учиться. Это честно указано везде, где приводятся цифры.

**— Что значит «GNN проигрывает — и это нормально»?**
В ML часто простой метод с умными признаками бьёт сложную нейросеть. Мы показали
обе модели, честно сравнили и объяснили, как усилить нейросеть дальше. Скрыть
проигрыш и показать только красивое — было бы жульничеством.

**— Как всё запустить, я не программист?**
Установите Docker Desktop, затем в папке проекта: `docker compose up -d`.
Через минуту откройте http://localhost:8501 — увидите демо с выпадающим списком
счетов. Выберите верхний (самый подозрительный) — увидите цепочку и объяснение.
